# CLAUDE.md

## Repo Purpose

Playwright TypeScript test boilerplate. Two independent test suites with separate configs:

- **UI e2e**: `tests/*.spec.ts` — runs with `playwright.config.ts`, uses 3 browsers
- **API**: `tests-api/*.test.ts` — runs with `playwright.api.config.ts`, uses the `request` fixture

## Commands

```
yarn test          # run all UI tests (chromium, firefox, webkit)
yarn test:api      # run API tests
yarn test:ui       # open Playwright UI mode
yarn test:ci       # CI run with sharding
yarn lint          # lint check
yarn lint-fix      # auto-fix lint issues
```

## Environment Setup

1. Copy `.env.example` to `.env`
2. Set `USER_PASS` to the saucedemo.com password

The auth storage state is generated by `tests/auth.setup.ts` and saved to `playwright/.auth/standard-user.json`. This happens automatically before UI tests run (it is a dependency project in `playwright.config.ts`).

## Key Conventions

**Never deviate from these:**

- **Selectors**: always use `getByTestId('...')` — `testIdAttribute` is set to `data-test` in `playwright.config.ts`
- **Page objects**: extend `BasePage` from `lib/pages/base.page.ts`; locators must be `private readonly`; methods must be `async`
- **Components**: accept a `Locator` (not `Page`) in constructor; used for reusable sub-sections of a page
- **Test imports**: import `test` and `expect` from `@fixtures` (path alias for `lib/fixtures.ts`), not from `@playwright/test`
- **Unauthenticated tests**: add `test.use({ storageState: { cookies: [], origins: [] } })` at the top of the spec
- **UI constants**: import from `constants.ts` (base URL, user credentials)
- **API constants**: import from `constants-api-tests.ts` (API base URL)
- **API types**: define in `lib/types/api.types.ts`

## File Placement Rules

| What              | Where                              |
| ----------------- | ---------------------------------- |
| New UI spec       | `tests/feature-name.spec.ts`       |
| New API spec      | `tests-api/resource-name.test.ts`  |
| New page object   | `lib/pages/name.page.ts`           |
| New component     | `lib/components/name.component.ts` |
| New shared type   | `lib/types/api.types.ts`           |
| New shared helper | `lib/utils/name.ts`                |

After adding a new page or component, export it from the corresponding `index.ts` barrel file.

## Config Mapping

- `playwright.config.ts` → runs `tests/` → includes the `setup` auth project → uses `playwright/.auth/standard-user.json`
- `playwright.api.config.ts` → runs `tests-api/` → no auth project → uses the `request` fixture directly
- Never add UI specs to `tests-api/` or vice versa

## Anti-Patterns

- Do not use CSS class selectors like `locator('.some-class')` — use `getByTestId()` only
- Do not use `page.waitForTimeout()` — use auto-waiting Playwright assertions
- Do not import `test`/`expect` from `@playwright/test` in UI spec files — import from `@fixtures`
- Do not leave `test.only` in committed code (blocked by `forbidOnly` in CI)
- Do not put constants inline in test files — add them to `constants.ts` or `constants-api-tests.ts`
